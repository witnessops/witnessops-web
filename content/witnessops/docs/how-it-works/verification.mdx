# How to Verify a Receipt

This page answers one practical question: how to verify a receipt or bundle offline, and how to read `valid`, `invalid`, and `indeterminate` results without claiming too much.

## 1. Problem this page solves

Many teams can export artifacts but cannot say which checks were actually proven.

Verification needs a deterministic sequence that separates:

- proof that passed
- checks that were missing
- trust dependencies that still sit outside the bundle

This page defines that sequence.

## 2. What you should understand after reading

After this page, you should understand:

- the minimum artifacts needed to verify a receipt
- the exact order of checks for independent verification
- what each verdict means (`valid`, `invalid`, `indeterminate`)
- what is proven and what still depends on trust

## 3. Minimum artifact set

At minimum, start with the receipt material and trusted key material.

| Artifact | Status | Why it matters |
| --- | --- | --- |
| Receipt JSON or exported bundle | required | Core proof statement and references |
| Signing public key or trusted key source | required | You cannot check signature authenticity without it |
| RFC 3161 timestamp token + chain | conditional | Required when the receipt claims trusted time |
| Log checkpoint + inclusion proof | conditional | Required when the receipt claims append-only publication |

## 4. Verification procedure (exact order)

1. **Confirm the artifact set.** Determine whether you have receipt-only material or a fuller bundle.
2. **Verify manifest integrity.** If `MANIFEST.json` exists, recompute the listed hashes first.
3. **Verify the signature.** Check the signed payload against trusted key material.
4. **Verify structure and bindings.** Confirm schema and digest references are internally consistent.
5. **Verify the timestamp layer.** If present, validate the RFC 3161 token and chain.
6. **Verify the log inclusion layer.** If present, validate the inclusion proof and checkpoint signature.
7. **Verify continuity links.** If prior-state links exist, confirm there is no chain break.

Do not skip the order. Integrity and signature checks come before you interpret higher proof layers.

## 5. Expected results and interpretation

| Check | Outcomes |
| --- | --- |
| Bundle integrity | pass / tampered |
| Signature | valid / invalid / missing key |
| Structure and binding | well-formed / malformed / digest mismatch |
| Timestamp layer | valid / invalid / not present |
| Log inclusion layer | valid / invalid / not present |
| Continuity layer | proven / not present |

Verdict interpretation:

- **`valid`**: the required checks for the declared claim set passed.
- **`invalid`**: one or more proof-bearing checks failed.
- **`indeterminate`**: the bundle may be coherent, but a required outside trust condition could not be established locally.

## 6. What is proven vs what remains trusted

**Proven when checks pass**

- issuer-bound signed statement integrity
- digest and artifact consistency
- optional trusted-time and inclusion claims when the matching material is present and valid

**Still trusted externally**

- key distribution authenticity
- TSA trust and time-source trust
- log-operator trust model, unless additional witness policy is satisfied

## 7. Common failure modes

- Missing file or hash mismatch → tampered or incomplete bundle.
- Signature failure → issuer authenticity not established.
- Timestamp failure → trusted-time claim not established.
- Inclusion proof failure → append-only publication claim not established.

Report failures by layer instead of collapsing everything into one vague pass/fail line.
