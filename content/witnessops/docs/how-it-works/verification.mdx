# How to Verify a Receipt

This page solves one operational question: how to verify a receipt or bundle offline, and how to interpret `valid`, `invalid`, and `indeterminate` results without overstating claims.

## 1. Problem this page solves

Many teams can export artifacts but cannot explain which checks were actually proven.

Verification requires a deterministic sequence that separates:

- cryptographic proof that succeeded
- checks that were not present
- trust dependencies that remain external

This page defines that sequence.

## 2. What you should understand after reading

After this page, you should understand:

- minimum artifacts required to verify a receipt
- exact check order for independent verification
- what each verdict means (`valid`, `invalid`, `indeterminate`)
- what is proven vs still trusted

## 3. Minimum artifact set

At minimum, start with receipt material and trusted key material.

| Artifact | Status | Why it matters |
| --- | --- | --- |
| Receipt JSON or exported bundle | required | Core proof statement and references |
| Signing public key / trusted key source | required | Signature authenticity cannot be checked without it |
| RFC 3161 timestamp token + chain | conditional | Required when trusted-time claim is present |
| Log checkpoint + inclusion proof | conditional | Required when append-only publication claim is present |

## 4. Verification procedure (exact order)

1. **Confirm artifact set.** Determine whether you have receipt-only or a fuller bundle.
2. **Verify manifest integrity.** If `MANIFEST.json` exists, recompute listed hashes first.
3. **Verify signature.** Check signed payload against trusted key material.
4. **Verify structure and binding.** Confirm schema and digest references are internally consistent.
5. **Verify timestamp layer.** If present, validate RFC 3161 token and chain.
6. **Verify log inclusion layer.** If present, validate inclusion proof and checkpoint signature.
7. **Verify continuity links.** If prior-state links exist, ensure no chain break.

Do not skip sequence order. Integrity and signature checks come before interpreting higher proof layers.

## 5. Expected results and interpretation

| Check | Outcomes |
| --- | --- |
| Bundle integrity | pass / tampered |
| Signature | valid / invalid / missing key |
| Structure and binding | well-formed / malformed / digest mismatch |
| Timestamp layer | valid / invalid / not present |
| Log inclusion layer | valid / invalid / not present |
| Continuity layer | proven / not present |

Verdict interpretation:

- **`valid`**: required checks for the declared claim set passed.
- **`invalid`**: one or more proof-bearing checks failed.
- **`indeterminate`**: structure/signature may be plausible, but required external trust condition could not be established locally.

## 6. What is proven vs what remains trusted

**Proven when checks pass**

- issuer-bound signed statement integrity
- digest and artifact consistency
- optional trusted-time and inclusion claims when corresponding material is present and valid

**Still trusted externally**

- key distribution authenticity
- TSA trust and time-source trust
- log-operator trust model (unless additional witness policy is satisfied)

## 7. Common failure modes

- Missing file or hash mismatch -> tampered or incomplete bundle.
- Signature failure -> claimed issuer authenticity not established.
- Timestamp failure -> trusted-time claim not established.
- Inclusion proof failure -> append-only publication not established.

Report failures by layer rather than collapsing everything into a single vague pass/fail statement.
