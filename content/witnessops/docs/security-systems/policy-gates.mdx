# Policy Gates

Policy gates are the runtime enforcement layer that turns authorization policy into an allow-or-block decision before a step executes.

## 1. Problem this page solves

Authorization can look complete on paper while execution still runs outside intended boundaries.

This page defines how WitnessOps enforces preconditions at runtime so policy is not treated as documentation-only.

## 2. What you should understand after reading

After this page, you should understand:

- which gate families are evaluated before execution
- how gate composition and evaluation order work
- what happens when a gate fails
- what gate evidence is emitted for later review

## 3. Enforcement mechanism

Gates are declared with runbook steps and evaluated at execution time.

For a step to run, all required gates for that step must pass. Evaluation is ordered and short-circuits on first failure.

That design prevents “partial approval” behavior and preserves deterministic reviewability.

## 4. Gate families

| Gate family | What is enforced | Failure meaning |
| --- | --- | --- |
| Scope check | Target stays inside approved scope boundary | Out-of-scope execution attempt blocked |
| Approval requirement | Required approver has explicitly authorized step | No valid approval path |
| Target authorization | Target has pre-authorization for engagement/context | Target not authorized for this run |
| Environmental constraint | Runtime context satisfies policy window/classification/network constraints | Right action, wrong context |
| Freemail rejection | Principal identity meets governance identity requirements | Identity quality below policy floor |
| Tool allowlist | Step tool is permitted for workflow risk class | Unauthorized capability invocation |

## 5. Gate failure semantics

When a gate fails:

1. the step does not execute
2. failure reason and failing gate are recorded
3. operation either pauses for correction or ends if terminal
4. receipt evidence captures the blocked state

A blocked gate is a successful control outcome, not a runtime defect.

## 6. Evidence and trust boundary

Receipts should record gate set, evaluation order, pass/fail states, and approval identity/timestamps where required.

This proves policy checks were evaluated as part of execution path.

What remains trust-based:

- correctness of external policy sources
- correctness of upstream identity systems
- quality of scope inputs and classification mapping
