# Proof Artifact Classes

This page is the artifact-class naming contract for operators, approvers and reviewers, and integrators. Use it to keep artifact names consistent across public docs, `/verify`, `/api/verify`, and retained-reference verifier or corpus material.

## 1. Problem this page solves

Artifact labels drift fastest at the boundaries between execution, verification, and review. This page prevents drift by defining the current class names, ownership authorities, status labels, and where each class is used.

## 2. Who uses this reference and expected outcome

After this page:

- operators can map run outputs to the correct artifact-class names
- reviewers and approvers can separate live and public classes from retained-reference-only classes
- integrators can avoid promoting internal or corpus-only terms into public contracts

## 4. Status semantics and verification-result semantics

### Artifact-class status labels

| Status label | Meaning | Interpretation boundary |
| --- | --- | --- |
| `live / public` | Current class name for public docs and operator or reviewer-facing surfaces | Safe to use in current public documentation and route-level explanations |
| `live output term` | Current public output vocabulary from verification-facing surfaces | Use for result reporting; do not expand beyond stated verification scope |
| `current verifier / bundle term` | Active term inside bundle-verification structure | Treat as bundle-contract vocabulary, not a generic runtime-event label |
| `retained-reference / corpus class` | Observed in retained-reference verifier or corpus material | Do not present as guaranteed live or public class behavior |
| `planned, not live` | Name appears in internal or planning lanes but is not frozen publicly | Must be marked non-live; not a public contract claim |

### Verification-result status semantics

For verification output interpretation:

- `valid`: required checks for the declared scope passed
- `invalid`: one or more proof-bearing checks failed
- `indeterminate`: an external trust requirement was declared but could not be established locally

Current `/api/verify` v1 for supported receipt inputs is receipt-first and returns deterministic receipt-lane verdicts, while malformed or unsupported input classes are reported separately (`FAILURE_INPUT_MALFORMED`, `FAILURE_INPUT_UNSUPPORTED`).

## 5. Explicit scope limits and non-claims

This page is a naming and status reference only. It does not:

- freeze a full, universal proof-object ontology
- widen `/verify` v1 into full bundle-upload or artifact-byte revalidation claims
- promote retained-reference or corpus-only terms into live or public operator contracts
- replace [Verification](/docs/how-it-works/verification), [Receipt Spec](/docs/evidence/receipt-spec), or [Threat Model](/docs/security-systems/threat-model)
- expose internal-only proof implementation detail through operator-facing surfaces
